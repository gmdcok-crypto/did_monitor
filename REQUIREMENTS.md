# DID Monitor — 요구사항·맥락 정리

> `did` 본체와 분리된 **운영/모니터링** 프로젝트. 로컬: `D:\did_monitor`, GitHub: `did_monitor` 레포, Railway: `did_monitor` 프로젝트 + MySQL.

---

## 1. 목적

- **여러 고객(여러 DID 배포)** 이 늘어나도 **한눈에** 파악할 수 있는 화면.
- **DID 앱과 직접 연관된 것만** 범위로 둠 (전사 Cloudflare·전체 Railway가 아님).

---

## 2. 한 화면에서 보고 싶은 것 (메타)

| 항목 | 설명 |
|------|------|
| Railway | 고객별 **프로젝트/서비스** 식별(이름·ID 등) |
| URL | DID가 서비스되는 **공개 URL** (예: `*.up.railway.app` 또는 커스텀 도메인) |
| R2 | **버킷 이름**, **공개 베이스 URL** (`R2_PUBLIC_BASE_URL` 수준). 비밀키는 UI/DB 평문 저장 지양 |
| MySQL (DID 쪽) | 참고용으로 **DB 이름** 정도(예: `railway`). 접속 비밀번호는 저장소·채팅에 넣지 않기 |

---

## 3. 나아가서 (집계·모니터링)

- **고객(인스턴스)별 디바이스 수**, 온라인/오프라인 수 등 **종합 지표**.
- 데이터는 **각 DID 인스턴스의 DB**에 있으므로, 중앙 앱이 **각 DID API를 호출**하거나 **주기적 동기화** 같은 설계가 필요함.

---

## 4. 제어 기능 (선택·추가 가능 범위)

- **서비스 중지 / 재시작** 등: **Railway API**(또는 CLI 자동화) 연동으로 가능한 경우가 많음.
- **주의**: 강한 권한 토큰 필요 → **최소 권한**, **운영자만 접근**, **감사 로그**(누가·언제·무엇을).

---

## 5. R2 사용 용량

- DID가 쓰는 **해당 버킷** 기준으로, **Cloudflare API/대시보드**에서 조회해 표시하는 방식이 일반적.
- 구현 시점의 **공식 API 문서**에 맞춰 엔드포인트·권한 확인.

---

## 6. 인프라 구성 (합의된 방향)

| 구분 | 내용 |
|------|------|
| 저장소 | GitHub **`did_monitor`** ( `did` 와 별도 ) |
| 로컬 폴더 | **`D:\did_monitor`** 등 **did와 다른 폴더** |
| Railway | **`did_monitor` 전용 프로젝트** — DID 고객 인스턴스와 역할 분리 |
| DB | 모니터 앱용 **MySQL** (예: 플러그인 `MYSQL_URL`, DB명 `railway` 등 플랫폼 기본값 가능). **등록 고객·스냅샷·감사 로그** 저장 |
| DID 고객 | **고객마다 DID는 별도 Railway 프로젝트 + 별도 DB + (권장) 별도 R2 버킷** — 앱 코드는 같고 연결만 분리 |

---

## 7. 보안·운영 메모

- DB **루트 비밀번호**, API 키, Railway 토큰은 **채팅·스크린샷·Git 커밋에 넣지 않기**. 노출 시 **회전(재발급)**.
- `did_monitor` DB에는 **비밀번호 평문 저장 지양**; 토큰은 환경 변수·Railway Secrets.

---

## 8. Cursor / 작업 폴더

- **`did` 폴더만 열고 채팅**한 내용은 **`did_monitor` 폴더를 열면** 같은 스레드가 안 보일 수 있음. 중요한 결정은 **이 파일처럼 레포에 남길 것**.

---

## 9. 다음 구현 단계 (참고)

1. `did_monitor` 레포에 **백엔드 + 프론트**(또는 풀스택) 스택 결정  
2. Railway 서비스에 **Git 연결**, `MYSQL_URL` 등 변수 연결  
3. DB 마이그레이션: **고객 등록 테이블**, (선택) **지표 스냅샷·감사 로그**  
4. 각 DID 인스턴스용 **읽기 전용 요약 API**(또는 기존 API 활용) 설계  
5. (선택) Railway·Cloudflare API로 **재시작·용량 조회**
